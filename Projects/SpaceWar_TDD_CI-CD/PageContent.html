Description

CI-CD & TDD

Continuous Integration and Continuous Deployment (CI/CD) automates the process of validating, building and distributing a project after every change.

The goal is to reduce manual work, detect issues earlier and guarantee that every version of the project is built in a reproducible environment.

Test-Driven Development (TDD) is a development methodology where tests are written before implementing the actual feature.

The workflow follows a short iteration cycle: write a failing test → implement the code required → refactor while keeping tests passing.

Content


Unity integration is handled through GitHub Actions workflows.
The pipeline is integrated directly with GitHub so source control becomes the entry point of the automation workflow.
Every push triggers GitHub Actions, which executes the CI pipeline inside an isolated environment and performs the build process automatically.

The pipeline fisrt installs the required Unity Editor version in batch mode to avoid opening the editor. The editor version and dependencies are fully defined as code inside workflow files stored alongside the project. This guarantees every build runs with identical configuration regardless of the machine executing it.

The pipeline then tries to build the project. On build failure or test failure, the execution is immediatly aborted. This ensures that code on branches protected by the pipeline always works (produces a playable build) and is also compliant with the initial rules (set by the tests). This ensures that no push accidentally breaks another feature without having to manually test everything manually.
Though the set up might be cumbersome at first, it ends up greatly reducing the manual workload and the reliability of the product which is definitely worth it in the end.


Here s how it s implemented.

Unity requires a license to use and you therefore have to provide some informations for Github to use ur license. This is done through Github secrets (can be done through repository's settings tab). The secrets to provide are:
- UNITY_EMAIL (email used for your unity account)
- UNITY_PASSWORD (password used for your unity account)
- UNITY_LICENSE (your license's content)

The next step is to setup the Unity project.

The game is contained within a Game folder with its own Assembly definiton and the code is in a Tests folder with another Assembly definition. The Tests folder ensures Unity's build system doesn't pick it up in builds and the assembly definitions ensure it's kept in separate project (.csproj) and compiled independently.

As TDD intends, we first have to write the tests. 


using NUnit.Framework; // framework used for testing


[TestFixture]
public class PlayerTests
{
    private Player player;

    [SetUp]
    public void SetUp()
    {
        player = new Player();
    }

    [Test]
    public void TakeDamage_Normal_ReducesHealth()
    {
        int damage = 20;
        player.TakeDamage(damage);

        Assert.AreEqual(player.BaseHealth - damage, player.Health);
    }

    ...
}

Code explanation

[TestFixture] -> Attribute marking the class for the pipeline to pick it up [SetUp] -> Attributes marking the method as the method to initialize the class before running each Test [Test] -> Attributes marking the method as a test method Assert -> Static class that defines wether the test succeeded or failed. If the assertion isn't correct the test has failed

The SetUp() method makes sure the test is run on a untouched instance of the class.
The TakeDamage_Normal_ReducesHealth() method make sure the player actually takes damage when the TakeDamage method is called. The naming convention is MethodItsTesting_Scenario_ExpectedResult.
We intentionally avoid hardcoded/magic values to avoid issues because they are error prone and often rely on assumptions (example on why below)

We can now write the code to fulfill the test. 


public sealed class Player
{
    public const int BaseHealth = 100;
    public int Health { get; private set; } = BaseHealth;

    public void TakeDamage(int amount)
    {
        Health = Mathf.Max(0, Health - amount);
    }

    ...
}

This minimal implementation passes the test however the only test we implemented is not sufficient are there are still some flaws. For instance, dealing negative damage would heal. if that is intended behaviour, then great but it should be tested nevertheless. That is why we developp an entire set of test to test edge-cases. 


// PlayerTests.cs
[Test]
public void TakeDamage_WithNegativeAmount_DoesNotChangeHealth()
{
    player.TakeDamage(-10);

    Assert.AreEqual(player.BaseHealth, player.Health);
}
// this simple test highlights that issue and forces us to adapt the code

// Player.cs
public void TakeDamage(int amount)
{
    if (amount <=  0)
        return;

    Health = Mathf.Max(0, Health - amount);
}

Any modification of the method will still need to comply with the tests and therefore avoid breaking behaviour.

This is why TDD and CI-CD are great. We write tests according to the intended behaviour.
We then write the code to comply with these tests and therfore with the intended behaviour.
Then, the CI-CD pipeline runs the tests and protected your code from breaking.
