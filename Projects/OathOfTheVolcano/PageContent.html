Description

Oath Of The Volcano

This is a small roguelike made with Unreal Engine for a school project (4 weeks). The rules where { 60s games, First person, Fast-paced gameplay }; We decided to make a game in which the player is locked in a 60s time loop and has to beat the boss within that time window. The player can find chests scattered around the map to get stronger. Every stat increase is kept on death or end of the 60s cycle

Content


The game flow architecture is built around centralized singleton managers combined with Blueprint Interfaces and Event Dispatchers to keep gameplay systems decoupled and synchronized.

A Blueprint Interface named IGameFlowAffected defines the contract shared by all gameplay systems reacting to game flow events.

The interface exposes: OnPlayerSpawn(), OnGameReset(), OnGamePaused() and OnGameResumed().

Gameplay entities implement the interface to initialize when the player spawns, reset their internal state on cycle restart and stop/resume their logic during pause events.

MainGameInstance acts as the highest-level runtime manager. Because GameInstance persists for the entire application lifetime, it is used as a stable singleton responsible for player lifecycle management and global game flow initialization.

When the player is spawned or removed, Event Dispatchers broadcast the event to all subscribed systems. This Observer pattern avoids direct dependencies between gameplay systems.

MainGameMode handles gameplay-cycle logic such as resets and the pause system. The pause system monitors player input to toggle pause and resume states while the reset system is handled by MainGameMode itself. All of which is then propagated to relevant gameplay systems using the Observer pattern. Since GameMode exists only once per match, it acts as the authoritative gameplay singleton responsible for synchronizing runtime events.

The reset system uses SetTimerByEvent running in a loop to drive gameplay cycles. Once triggered, the GameMode broadcasts reset events through Event Dispatchers so every affected gameplay entity can adjust its state independently.

The game loop makes use of interfaces to enforce proper compliance for every actor relevant to the system and Event Dispatchers (Observer Pattern) to keep the code clean and avoid dependencies This architecture allows gameplay systems to be properly managed without enforcing dependencies onto the MainGameMode.

The combat system uses an interface-driven architecture to decouple hit detection from the specific actors being hit. Any actor can become damageable by implementing the IHittable interface without modifying the combat system itself.

The interface defines a standardized contract containing: void OnHit(float Damage, FVector Direction, Player Instigator) and bool IsRelevantToComboSystem().

When the player attacks, an attack animation is played and the weapon hitbox is activated during the appropriate animation frames.

Every overlapping actor is checked for the IHittable interface. If the actor implements the interface, the combat system queries IsRelevantToComboSystem() to determine whether the hit should contribute to the player's combo count.

This allows certain actors such as enemies to increase combo progression while ignoring objects that should still receive damage but not affect scoring.

Once validated, the combo counter is updated and OnHit() is invoked on the target. The target remains responsible for handling its own damage, reactions and gameplay effects.

This follows the Open/Closed Principle: new damageable actors can be added by simply implementing IHittable, without requiring any modification to the combat system itself.

Using an interface (IHittable) allows any actor to become hittable by simply implementing two methods, keeping the combat system open for extension and closed for modification.

It decouples the combat system from concrete actor classes as any actor implementing can react to hits an optionnaly participate in the combat system.

This interface driven architecture decouples combat logic from actor classes which makes the system scalable and extensible, new ennemies or object can join the system easily. If any change is needed, it can be easily done by modifying the interface and adjust the actors implementing it as needed. The interface enforces compliance and therefore any actor class implementing it won t be able to compile before it actually implements every member of the interface.

The key rebinding system is generated dynamically from the player's saved input settings. Instead of manually creating UI entries for every action, the menu is built directly from the registered input mappings, ensuring the interface always reflects the current control scheme.

The system retrieves the player's input settings, extracts all key mapping rows and iterates through them to discover every rebindable action.

For each mapping found, a dedicated Rebind Button widget is instantiated and added to the rebinding menu automatically.

Each Rebind Button encapsulates the data required to modify a specific mapping. This keeps the menu generic and prevents the parent widget from containing action-specific rebinding logic.

The widget consists of three elements: a label displaying the action name, a button used to capture a replacement key and a reset button used to restore the default binding.

Key rebinding is handled through Unreal's Input Key Selector widget. When activated, the widget temporarily listens for the next key press and automatically returns the selected input.

The new key is then applied to the corresponding mapping and the displayed value is refreshed to reflect the updated binding.

This design follows a data-driven approach where the UI is generated from player settings rather than hardcoded layouts. Adding new actions requires no UI modifications, making the system scalable and easy to maintain.
