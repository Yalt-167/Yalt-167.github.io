Description

This is a POC I made to learn how to use UGS (Unity Game Services) along with Netcode for GameObjects with Unity

The game flow is pretty simple. Upon connection, the user is greeted with a page to login to Unity Services (This part would be automatized in a real game but the backend relying on Steam, I wasn t gonna create a full Steam Project so this will be good enough for now)

Once connected to UGS, The user can decide to host a party or join one. This system makes use of the Unity services "Lobby" for the party and "Relay" for the routing. The wanted gamemode and map and lobby settings (size, accessibility, name, etc) can be adjusted as wanted.
When the lobby is populated as wanted, the host can launch the game. The players choose their team and the game can start

Multiplayer lobby system built with Unity Lobby, Relay and Netcode for GameObjects. The architecture centralizes all matchmaking logic inside a single LobbyHandler singleton to keep UI, relay setup and lobby synchronization coordinated. 


public sealed class LobbyHandler : MonoBehaviour
{
    public static LobbyHandler Instance { get; private set; }

    private Lobby hostLobby;
    private Player localPlayer;
}

A Singleton pattern is used so every gameplay or UI system can access lobby state without repeatedly searching scene references. The class is marked sealed to prevent unintended inheritance on a critical networking manager.

Unity Authentication is initialized asynchronously before any matchmaking operation. Sign-in callbacks keep the local session state synchronized with Unity Services events. 


await UnityServices.InitializeAsync();

AuthenticationService.Instance.SignedIn += SignInCallback;
AuthenticationService.Instance.SignInFailed += SignInFailedCallback;

Lobby metadata uses indexed DataObjects so matchmaking filters remain server-side and scalable. Game mode, map and region can be queried efficiently without downloading every lobby. 


Data = new Dictionary<string, DataObject>()

Heartbeat and polling systems run independently to keep lobbies alive while minimizing unnecessary requests. Timers are updated manually instead of relying on coroutines for tighter control over network refresh rates. 


private async void HandleHeartbeat()
{
    if (!IsLobbyHost())
        return;

    heartbeatTimer += Time.deltaTime;

    if (heartbeatTimer >= heartbeat)
    {
        heartbeatTimer = 0f;
        await LobbyService.Instance.SendHeartbeatPingAsync(hostLobby.Id);
    }
}

Relay and local transports are abstracted behind dedicated selection methods. This allows the same gameplay code to run either through Unity Relay or local LAN testing without changing higher-level systems.

Once players have gathered inside a lobby, they must be assigned to teams before the match can begin. So far they where The Team Selector acts as a synchronized pre-game staging system, ensuring every client sees the same team composition before gameplay starts.

Networking is implemented using Unity Netcode for GameObjects RPCs. Rather than manually serializing packets, network behavior is expressed through standard C# methods decorated with RPC attributes.

Methods marked with [Rpc(SendTo.Server)] automatically execute on the server when called by a client, while [Rpc(SendTo.ClientsAndHost)] broadcasts execution from the server to every connected player.

Under the hood, Unity's Netcode analyzers and code generation tools detect RPC attributes during compilation and automatically generate the networking layer required to serialize parameters, route messages and invoke the target method on the appropriate machine.

This allows networked gameplay logic to remain readable and close to ordinary C# code while still executing across the network.

The execution flow follows a server-authoritative model: clients submit requests, the server validates them, and approved changes are broadcast back to every participant. This ensures all players share a consistent view of team assignments before entering the match.

The system is implemented as a networked singleton because only one team selection state should exist for the entire match. Centralizing team assignment prevents conflicting team data and provides a single authoritative entry point for all team-related operations.

Team configuration is initialized by the host and synchronized to every connected client. The number of teams and maximum team size are distributed through RPCs, allowing each client to locally construct an identical representation of the team selection screen.

Team membership is stored as lightweight arrays of player names. Since the system only exists during the pre-game phase, a simple array-based structure provides predictable memory usage and fast iteration while avoiding unnecessary complexity.

Players can freely switch teams through the menu. When a new team is selected, the player is automatically removed from their previous team before being inserted into the new one, ensuring team membership remains consistent across all clients.

The match can only begin once every connected player has joined a team. This validation prevents incomplete team assignments and guarantees the game starts from a valid state.
